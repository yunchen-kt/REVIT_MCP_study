SPECTRUM DECISION A / B / C / D

MCP 給 0/1
設計師走光譜

MCP 回 PASS / FAIL 就好。FAIL 不是死刑——它是四條路線的決策起點。本頁定義 A/B/C/D 框架，套到採光 / 排煙兩個典型 FAIL 場景，並從本場日本 sample 模型 demo 跑出的實際 FAIL 案例庫出發。

SECTION 01 · 兩者角色截然不同

MCP vs 設計師 — 不同的工作模式

MCP 是 執行器，回答「達標／不達標」。設計師是決策者，回答「能接受嗎？怎麼修？」。把這兩種模式混淆是大多數「AI 取代設計師」恐慌的源頭。

MCP（執行器）

  • PASS / FAIL（二元）
  • 0/1 結果，無灰色地帶
  • 硬如鋼——法規條文的字面執行
  • 對應 P13 MCP 是 0/1 執行器
  • 不負責推理「退一步海闊天空」
VS

設計師（決策者）

  • 「能接受嗎？怎麼修？」（光譜）
  • 維度與漸層的考量
  • win-win strategy，找最少退讓
  • 對應 P11 光譜在設計師
  • 負責整合美學 / 法規 / 業主期待

互不替代

把 MCP 的 0/1 結果硬塞「決策建議」（如「我建議您改成 1.2 m」）= 越界。設計師收到 FAIL 後，自己選 A/B/C/D 路線。MCP 只負責清楚說「未達 X 標準，差 Y 數值」。剩下交給人。

SECTION 02 · 通用決策框架

FAIL 之後 · 四條路線

不管是採光、消防、容積還是停車，FAIL 之後的決策都收斂到同一組四選一。框架是 A 改設計 / B 接受現狀 / C 改定義 / D 跳出 AI 找人。

A
改設計（直接修）
擴大窗、加開排煙窗、改走管路徑、加柱⋯⋯把 model 改到 PASS。最直觀，但要評估連動成本（結構 / 立面 / MEP 衝突）。
COST：連動評估必跑 · 適用：早期階段 / 衝突項少
B
接受現狀（不改）
承認「法規這樣寫，沒得改」。把 FAIL 留下 紀錄/，作為已知的不達標項繼續推進。需要主管簽字 + 業主同意。
COST：需要文件背書 · 適用：合規空間極小 / 後續審查可商榷
C
改定義（改房間用途）
把「臥室」改成「儲藏室」就不用查採光、把「居室」改成「非居室」就不用查排煙。合法繞道，但若實質用途仍是臥室，業主可能反對。
COST：業主接受度成本 · 適用：定義有彈性 / 業主可協商
D
跳出 AI（找人腦力激盪）
A/B/C 都卡死。AI 認輸——找老闆、業主、結構技師、消防顧問開會。對應 P12 死局求助。這是好事，不是失敗。
COST：時間成本，但避免錯路 · 適用：跨領域衝突 / 法規與業主期待矛盾
「AI 認輸」是好事，不是失敗。
它讓對的問題到達對的人。

CASE 01 · 採光 FAIL

套例 1：採光 FAIL（建技 §41）

居室採光面積比 ≥ 1/8（≈ 12.5%）。某居室實測 9.5%，FAIL。同一個 FAIL，四條路線。

A
加大窗
把 1.5×1.2 m 的窗放大到 1.8×1.5 m，採光比拉到 13.2% PASS。
連動風險：(1) 結構柱距是否允許？(2) 立面節奏是否被打破？(3) 增加熱負荷影響空調設計？
PASS 路徑：直接 · COST：結構 / 立面 / 空調連動
B
接受不達標
寫進不達標項紀錄，送業主與主管簽字。若送審時審查單位可商榷（如鄰棟距離特殊），可能放行。
實務上很少走——大多數情況審查不會放。
PASS 路徑：審查放行 · COST：文件 + 風險
C
改用途為「儲藏室」
儲藏室不是居室，不適用 §41 採光要求。FAIL 自動消失。
業主接受度：若實質是讓人睡的房間，業主會反對 → 不行。若是真的當儲藏 → 可行。
PASS 路徑：定義變更 · COST：業主談判
D
跳出找人
業主堅持要當臥室，結構不允許大窗，立面也不能改 → 找建築主管 + 業主三方會議。可能性：天窗、採光井、把房間挪到棟內別處。
對應 P12「死局求助」。
PASS 路徑：團隊決策 · COST：會議時間，但避免錯方向
CASE 02 · 排煙 FAIL

套例 2：排煙 FAIL（建技 §101 / 消防 §188）

某居室開口距天花 < 80 cm 的有效面積不足，FAIL。同一個 A/B/C/D 框架，套上去動作完全一致——這就是框架的普適性。

A
加開排煙窗
在天花下 80 cm 範圍內加一扇開放式排煙窗。
連動：立面是否容納？是否需要電動開啟機構？
PASS 路徑：硬體增加 · COST：機構 + 立面
B
改裝機械排煙
放棄自然排煙窗路線，改裝機械排煙系統（與 B「接受」不同——這算 A 的變體：改設計）。法規允許替代手段。
連動：MEP 設備量、電力、維護成本。
PASS 路徑：替代手段 · COST：機電 + 長期維護
C
改用途為非居室
機械室 / 儲藏室 / 走廊都不算居室，不適用排煙要求。能不能改要看實質用途與業主期待。
PASS 路徑：定義變更 · COST：業主談判
D
跳出找人
A/B/C 都被擋（業主不允用途變更、立面不允加窗、預算不允機械排煙）→ 找消防顧問 + 業主重新評估全棟排煙策略，可能要改平面格局。
PASS 路徑：團隊決策 · COST：可能改格局

框架的普適性 · 已被驗證

採光與排煙是兩個完全不同的法規條文，但 FAIL 之後的決策路線結構一致。這意味框架可以推廣到：容積率 FAIL、停車位數 FAIL、走廊寬度 FAIL、樓梯淨高 FAIL⋯⋯所有 MCP 給的 0/1 結果，都用同一個 A/B/C/D 收斂。

本場日本 sample 模型實測 FAIL

5/18 demo 用 サンプル住宅集合.rvt 跑出的真實 FAIL，作為框架的活體範例。不是教科書例題——是實際模型跑出來的。

店舗 26 排煙 FAIL（建技 §101）

1F 店舗 26 經 check_smoke_exhaust_windows 跑出：「有效開口面積不足，FAIL」。套 A/B/C/D：

路線具體做法業主／法規 / 美學阻力
A店舗面外的玻璃櫥窗改一部分為排煙開放窗會打斷店舗店面節奏，業主可能反對
A'裝機械排煙店舗預算不一定接受長期電費
C把該空間改成「事務空間（非營業）」？業主用途不允
D找消防顧問與業主三方評估建議走 D，因 A/A'/C 都有明顯阻力

駐車場 27 採光 FAIL（建技 §41 — 若該空間定義為居室）

get_room_daylight_info 顯示駐車場 27 採光比僅 4.5%。但這裡有個關鍵釐清——駐車場本來就不適用居室採光要求。實況：tool 的 raw data 是對的，但第二憲法 Domain Method Compliance要求查 domain SOP 確認房間是否需檢核。

FAIL 不一定要走 A/B/C/D

有時 FAIL 是檢核被誤用（用居室標準檢核非居室）。第二憲法要求先過 domain SOP 確認「該不該檢」——這比 A/B/C/D 早一步。框架的應用順序是：
1️⃣ 第一憲法 — 設計師問了才檢
2️⃣ 第二憲法 — 用對 domain SOP 檢
3️⃣ A/B/C/D — 真的 FAIL 才開始選路線

WHY
為什麼要做這個框架？因為大多數設計師看到 MCP 給 FAIL 的第一反應是「糟糕，怎麼辦」——而不是「選哪條路線」。框架把模糊的焦慮變成清楚的四選一，恢復決策權給人。
HOW
收到 FAIL → 30 秒內把 A/B/C/D 全部列出來（不是評估，是列出）。
再用 5 分鐘檢查每條路線的連動成本與業主接受度。
若三條都被擋，不要硬撐——走 D，找人。對應 P12「死局求助」是主動動作不是被動投降。
TECHNIQUE
Four-Path Decision Framework · Failure Reframing · Cross-Domain Applicability · Human-in-Loop Escalation
CHECKLIST
  • FAIL 訊息是否清楚說明差什麼數值（不只 PASS/FAIL）？
  • 設計師是否被訓練過 A/B/C/D 框架（而不是只想到 A）？
  • 選 D 是否被視為正常動作（不是「無能」）？
  • 連動成本評估是否有簡單檢核表（不必全跑模擬）？
為什麼 D 最被低估
大多數人覺得「找人」=「我搞不定」=「能力差」。
真實情況相反：越資深的設計師，越早觸發 D。
因為他們知道有些問題不該獨自背。
這也是為什麼 junior 工程師 容易陷在 A 路線，硬改設計到天荒地老 → 最後 schedule 炸了。senior 工程師看到「業主期待 + 法規 + 結構」三方衝突就直接 escalate，把問題送到對的決策層。D 不是失敗，是專業判斷。MCP 把它寫進框架，是為了把這個資深經驗變成每個設計師都能用的標準動作。

延伸閱讀

理論基礎
  • InternalP11 光譜在設計師
    為什麼設計師走的是維度與漸層，而不是 0/1。
  • InternalP13 MCP 是 0/1 執行器
    為什麼 MCP 不應該替設計師做決策。
  • InternalP12 死局求助
    D 路線的命題本源——AI 認輸是好事。
業界對標
  • MethodStage-Gate · Go/Recycle/Kill
    A/B/C/D 框架的概念近親——Stage-Gate 的三元決策也是「執行 vs 決策」分離。
  • Industry$15-25K 案例
    為什麼 D 路線值得花會議時間——後期返工成本比會議貴 100 倍。
