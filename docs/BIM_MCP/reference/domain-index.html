DOMAIN INDEX 40 個 SOP / 5 分類

Domain · 知識層
方法寫一次，多個 Skill 共用

Domain 是知識本身——法規條文、SOP、公式、例外。Skill 是編排（19 個）。Tool 是原子執行（88 個）。知識不應該重複在每個 Skill 裡——這是本專案核心架構原則。本頁列 40 個 domain/*.md，依五大類彩色標示，每張 card 含觸發關鍵字與反查 Skill。

SECTION 01 · FRAME

Tool / Domain / Skill 三層關係

10Compliance · 法規 13Workflow · 模型 / 圖面 7Ops · QA / authoring 9MEP / 構件 / 穿梁 1References · 外部

Tool（88）

執行 Revit 查詢、建立、修改。所有具體資料必須來自工具回傳。

Domain（40）

保存 SOP、公式、例外與判斷流程。方法的單一來源。

Skill（19）

把 Domain 與工具串成工作流。不藏方法在 Skill body。

為什麼不把每個 Domain 都升 Skill
防火 SOP 同時被消防、走廊、合規三個 Skill 引用。
若每個 Skill 都複製一份 → 法規修一次要改三份。
Domain 獨立存在，知識複用、編排薄層。
這是對 Anthropic 官方 Skill 模型（references 放 Skill 內）的合理特化。詳見 skill-authoring-standard.md。

SECTION 07 · CONTRIBUTION

新增 Domain 的順序

WHY
Domain 是方法的單一來源。法規修訂只改一份 SOP，所有引用它的 Skill 自動跟上。
HOW
先寫 domain/*.md，再決定要不要建 Skill（不是每個 domain 都要 Skill）。最後跑 /claude-md-sync 驗證雙向一致。
TECHNIQUE
Knowledge / Orchestration Separation · Single Source of Truth · No Skill Inflation
CHECKLIST
  • 新增 compliance 主題 → 先建 domain/*.md，含 frontmatter
  • 需要 AI workflow 編排再建/更新 Skill
  • 更新 CLAUDE.md 觸發表（38 行的那張）
  • 更新此頁卡片（含 trigger + 反查 Skill）
  • 跑 /claude-md-sync 與 /qaqc 驗收

→ 完整貢獻範本（含 6 步驟停車場淨高範例）

本 SOP 解決什麼

建技規則 §41：居室採光面積 ≥ 該居室樓地板面積的 1/8（≈ 12.5%）。但 tool 回的 raw data 不直接給 PASS / FAIL——必須套 75cm 台度扣除公式 + 納入外牆門開口（含玻璃部分）。

核心公式

有效採光面積 = Σ（窗開口面積）+ Σ（外牆門有效面積）

外牆門有效高度 = HeadHeight − 750mm （扣 75cm 台度基準）

採光比 = 有效採光面積 / 居室樓地板面積，必須 ≥ 12.5%

常見錯誤（違規範例）

2026-04-23 違規案例
  • ❌ 排除所有外牆門（誤以為門不算採光） → Room 203 被低估 1.38 m²
  • ❌ 未套 75cm 台度扣除 → 結果偏高
  • ❌ 用 LM prior 補計算 → 違反第二憲法 Domain Method Compliance

正確流程

  1. 呼叫 get_room_daylight_info(level='LX') 取得房間 + 開口 raw data
  2. 讀 domain/daylight-area-check.md 完整 SOP
  3. 對每個外牆開口（含門）套 Effective Height = HeadHeight − 750mm（若 SillHeight < 750mm）
  4. 計算採光比、輸出 PASS / FAIL + 證據
深度
  • Constitution第二憲法 Domain Method Compliance
  • Filedomain/daylight-area-check.md

本 SOP 解決什麼

建技 §101 + 消防 §188：居室排煙窗在天花板下 80cm 範圍內的有效開口面積，必須 ≥ 該居室樓地板面積的 2%（或機械排煙替代）。

三個判定

  • 無窗居室判定：完全無對外開口 → 強制機械排煙
  • 無開口樓層判定：整層平面對外開口不足 → 需檢討避難動線
  • 有效面積計算：僅天花下 80cm 內的窗計入，其下方部分忽略

常見錯誤

tool 預設行為踩雷
  • ❌ createAnnotatedViews=true 預設會建 80+ 視圖污染專案
  • ✅ 5/18 demo 後改預設 false（commit 4975ceb）

正確流程

  1. 呼叫 check_smoke_exhaust_windows(level='LX', createAnnotatedViews=false)
  2. 讀 SOP 確認天花下 80cm 規則 + 機械排煙替代條件
  3. 若 FAIL → 套 A/B/C/D 框架
深度
  • Skill/smoke-exhaust
  • SpectrumA/B/C/D 套例 2

本 SOP 解決什麼

結構工程容許梁穿孔（套管穿梁），但有嚴格位置與大小限制避免削弱抗剪能力。三種材料各有規則：

  • RC（混凝土）：Zone A 禁開 / Zone B 限制 / Zone C 許可
  • SC（鋼梁）：腹板 web opening 規則、flange 嚴禁
  • SRC（鋼骨混凝土）：依 composite beam internal steel 規則

本 base SOP 共用部分

  1. 判定梁類型（RC / SC / SRC）
  2. 分派到對應細項 SOP（RC / SC / SRC）
  3. 共用輸出格式（穿孔位置 + 大小 + Zone 判定 + 建議）

本 SOP 解決什麼

MEP 管線 × 結構（CSA = Civil / Structural / Architectural）碰撞偵測。傳統 Navisworks 用 Solid-to-Solid 計算量大，本 SOP 採 Curve-to-Solid 策略加速（把管線視為 curve）。

三階段

  1. 偵測：對指定 view / level 跑 detect_clashes
  2. 視覺化：自動 colorize_clashes 染紅／橘／黃三級嚴重度
  3. 匯出：export_clash_report 產出可送業主的 PDF / Excel

觸發場景

  • 每個階段完成點（P7）後跑一次——對應業界 70% 返工可避免的統計
  • 圖紙快出前必跑——避免「圖出後才發現碰撞」
深度

本 SOP 解決什麼

當新貢獻者想新增功能時——該寫 Domain？該寫 Skill？該加 Tool？ 沒有 SOP 會導致「每個小需求都建一個 Skill」造成 metadata 膨脹。

三選一決策樹

  • 只有方法 / 法規條文 → 寫 domain/*.md，不建 Skill
  • 有跨多 tool 的編排價值 + 觸發頻率高 → 才建 Skill
  • 沒有對應 tool → 先在 issue tracker 開 RFC，再評估加 MCP tool

Skill 入門檻

  1. 協調 ≥ 2 個 tool（單 tool 不必開 Skill）
  2. 觸發關鍵字高頻（冷門關鍵字停在 Domain）
  3. 引用既有 Domain（不寫新知識，寫新編排）

反例

不該升 Skill 的常見誤判
  • ❌ 把單一 tool 的「使用範例」當 Skill
  • ❌ 把複製貼上的 SOP 包成 Skill body（應該引用 Domain）
  • ❌ 為了 demo 漂亮而建 Skill
