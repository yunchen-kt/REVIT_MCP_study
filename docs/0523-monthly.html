「越做越醜」初步觀察

這個說法在業界很常聽到。學生作品給人「感覺新鮮」、自己 10 年前的方案「很大膽」、但現在做的設計越來越保守、功能主義、不像當初想做的。

我不認為這是「失去美感」。我認為這是流程的時序錯誤——一個我們沒看清楚的問題。

MCP 剛好可以幫我們看清楚這個問題。接下來 60 分鐘，我會帶大家拆解它。

設計流程的時序錯誤

P3

典型場景

你畫了一個漂亮的 1F 平面——大窗、寬走廊、雅致樓梯，自己很滿意。然後拿給結構技師：「這道剪力牆不能拆。」拿給機電：「這個走廊塞不下消防管。」拿給法規：「樓梯淨高違反 §65。」

你修改。但每次修改都讓原本的美學退一步。等到能交圖了，剩下的就是「醜」。

這不是失去美感，是流程

美學沒消失。是流程把它磨掉了——因為限制是後期才出現，每次出現都迫使你退讓。

如果限制在前期就能看到呢？你可以在第一筆下去之前就避開、調整、找替代方案——美學被保住的可能性大很多。

「越做越醜」的根因

P3

學生為什麼「感覺新鮮」

學生的方案大膽，不是因為他們有更高的美學品味——是因為他們還沒被限制壓過。資深設計師的保守，是被一次次「限制反噬」訓練出來的條件反射。

但這條件反射是悲劇

它讓資深設計師在前期願景階段就先自我審查，先放棄了所有「可能被結構打回」的大膽想法。結果就是大家做出來的東西越來越像彼此，越來越保守、越來越功能主義。

解法不是「鼓勵大膽」

解法是讓限制變得早期可見——這樣設計師敢大膽，因為他知道限制是什麼、他知道哪裡有迴旋空間。MCP 的角色就在這裡。

傳統 BIM 工具的角色定位

P1P2

Revit / ArchiCAD / Rhino 都是「執行作業」工具

它們解決的問題是：「怎麼把腦中的方案畫出來」。畫得快、畫得準、畫得可協作。

但它們不告訴你方案撞了哪些限制

它們不會主動跳出來說「這道牆違反防火區劃」「這個窗台高於規範」「採光比不夠」。要查這些，你得自己手動跑各種檢核——而檢核工具又分散在不同軟體、不同 plug-in、不同 Excel 模板。

這個分類為什麼重要

因為它告訴你 MCP 不是 Revit 的替代品。MCP 是補上 Revit 沒解決的另一半問題——「限制顯現」。兩者協作，不是競爭。

MCP 突破「執行作業」的可能

P1P4

傳統工具沒解決的

  • 限制檢核分散，每次要重跑
  • 檢核結果是文字 / 表格，不在模型上呈現
  • 檢核工具不知道彼此的存在，無法 orchestration
  • 檢核結果不會自動回饋到下個設計動作

MCP 有機會做的

  • 一個 prompt 編排多個檢核並行
  • 結果直接 override 在模型元素上（紅綠色）
  • 96 個工具可以組合成劇本
  • 對話介面降低「我該查哪個」的決策成本

但「有機會」不等於「會做對」

下一節 Section 2 就是要把「該怎麼做才對」講清楚。包含兩個常見誤解：MCP 該不該主動？MCP 該不該給建議？

什麼是「限制顯現器」

P4

使用者原文（2026-05-18 第一輪論述）

我認為 MCP 就是快速讓這些「限制」顯現，來讓設計者在企圖保有的需求、美學、意象等極大化的拉扯上，給出最少的退讓，獲得到最美與自己最喜愛的那個成果。 — 使用者，2026-05-18

關鍵字解析

  • 「快速」：早期、即時，不是事後
  • 「顯現」：讓看不見的變看得見，但不替你決定
  • 「最少的退讓」：拉扯中找到讓步最小的方案
  • 「最美與自己最喜愛」：保住美學主權，不是接受平庸合規

跟「自動化」的差異

自動化是「我替你做」。限制顯現器是「我讓你看見」。決策權還在你。

人機分工新模型

P5P6

為什麼舊模型錯了

「AI 設計、人類審核」這個模型是 AI hype 帶來的——它假設 AI 能做「創意工作」，人類退到「品管」角色。

實務上這個模型崩潰：AI 生出的設計品味平庸（訓練資料是業界平均），人類審核又被資訊淹沒（要看 100 個方案）。

新模型的翻轉

「AI 揭露限制邊界，人類在限制可視化的前提下放大膽追美學」——AI 退到「限制顯現」這個高價值但低姿態的角色，人類重新成為設計主導。

哲學意義

這個翻轉把「設計師之所以為設計師」還給設計師。AI 不取代你的創意，AI 把「枯燥的限制檢核」這塊枯燥的工作做掉，讓你的創意有更多空間。

誤解 #1：MCP 不該主動

P8

使用者原文（2026-05-18 第二輪論述）

限制應該不是主動，而是在設計者製作工作的過程中⋯⋯在尚未到達階段完成點的時候，如果有過多的資訊進入或來討論，會造成設計者的思緒上陷入迴圈打轉，而無法完成工作！⋯⋯因此主動跳出限制我覺得很怪，被動反倒是個好的狀態。 — 使用者，2026-05-18

技術上可以「主動」 vs 設計上選擇「不做」

技術上 MCP 完全可以接 Webhook、watcher、可以做 fail-fast。但「能做」不等於「該做」——主動跳出會打斷設計師的「思維落地」過程。

業界類比

Martin Fowler 的 Fail-Fast Architecture 在軟體工程是好設計，但搬到「設計師創作過程」就是噪音。場域不同、要求不同。

誤解 #2：MCP 是 0/1 執行器

P13

使用者糾正我的原文（2026-05-18 第三輪論述）

我認為部分的檢討就是 2 元，而不是我講的光譜做法。光譜是解決辦法，MCP 是執行的結果，結果只有對錯，但不代表錯誤一旦被忽略則專案無法執行，這件事情你必須要先能夠同意。 — 使用者，2026-05-18

這條糾正非常重要

我前面建議「Domain 重構為光譜回應」是錯的。MCP 的本質就是「檢核出對/錯」，光譜思維是設計師接到結果後的人類決策，不是 MCP 該承擔的。

「對錯不代表無法執行」

很多送審圖也是「妥協後通過」的——FAIL 不是 GAME OVER，是設計師決策的起點。Domain 不需要重構為光譜回應，現有 0/1 設計是對的。

階段完成點概念

P7

使用者定義

這個完成點不是我們說的主計畫里程碑，大多是在它們思維或者想法上被首次落地的那個剎那，所以稱為階段完成。 — 使用者，2026-05-18

為什麼不能更早

在「想法尚未落地」之前，過多資訊會讓設計師思緒打轉。沒有具象成果可以被檢視，所有檢核都是抽象的——「假設你做 X 就會違反 Y」沒有實際幫助。

必須等想法落地——畫出一個具象方案——MCP 才有東西可以檢，設計師才有東西可以接收結果做決策。

怎麼判定「階段完成點」

這是命題 P16（Work Breakdown）的工作——把設計拆成小單元，每個單元有可交付物，每個交付物 = 一個階段完成點。

排程死線是設計流程的第一限制

P15

使用者原文

抵達死線下的完成度 80 和 60 和 100 的差異其實無差異，反而是 0 才是無法被判斷的狀態，那這樣的工作流程就可以開始轉動。 — 使用者，2026-05-18

業界對標：Stage-Gate 模型

Robert Cooper 1986 起源汽車業的 Stage-Gate Model 有三元決策：Go / Recycle / Kill——對應「完成度 100 / 80-60 / 0」。40 年產業驗證。

Agile 哲學的呼應

「Incompleteness is harnessed」（Zero Blockers）——不完整是策略，不是缺陷。Design-Build 正式承認「the architectural and engineering design cannot be complete」(Korte Company)。

實務含意

只要有交付物（不管完成度多少），下個審視動作就能啟動。0 才會卡死整條工作流。MCP 該支援這個哲學——能接受 80/60 完成度的模型也跑檢核，而不是堅持「沒做完不能查」。

Work Breakdown 讓階段點可識別

P16

沒拆分的後果

「我在做設計」——這是一個太大的工作單位。沒有節點，MCP 只能等「整塊完成」（可能要 1-2 週）才能介入。

拆分後

「1F 平面 LOD 200 完成」「外牆系統 LOD 300 完成」「樓梯 LOD 250 完成」——每個都是可被識別的階段完成點，每個都可以觸發 MCP 對應的檢查工具組。

業界標準

  • BEP (BIM Execution Plan) — ISO 19650
  • LOD Specification — AIA / BIM Forum
  • Work Breakdown Structure Guide

反模式警告：8/80 Rule

WBS 業界規則：工作包應 8-80 小時，過細變 micromanagement 反而拖垮。警告：96 工具不該變成 500 個微任務。

業界對標物：BEP / LOD / IDS

三個都成熟，但沒被串成工具鏈

BEP 是「該檢查什麼的計畫」、LOD 是「檢查的完成度標準」、IDS 是「機器可讀的檢查規格」。三件事業界都有，但沒人把它們串成一個一鍵觸發的工具鏈。

這就是「96 工具編排平台 = 業界真空白」的具體含義。我們不是發明新概念，是 orchestration 既有概念。

設計 = 解決問題，美學 = 手段

P10

使用者糾正原文

這件事情不該被強調。設計的真實重點叫做解決問題，美學是一個重要的手段，但是能夠解決各方問題同時保持美感的結果，就是設計者的功力考驗，因此沒有你說的悖逆項限制。 — 使用者，2026-05-18

糾正的對象

我前面提過「美學被限制反噬」的論述——使用者明確說這是假命題。設計師不是「美學主義者」，是「解決問題者」。能在解決問題時保持美感才是真功力。

對 MCP 設計的影響

MCP 不該假裝「在保護設計師的美學」——它的任務就是把限制呈現清楚，至於「保住多少美學」是設計師的本事，不是 MCP 的責任。

光譜決策 vs 0/1 執行

P11P13

實例

採光 FAIL。MCP 回「FAIL」+ 數據。設計師面對的光譜：

  • 選項 A：加大窗 → 結構衝突？立面美感衝突？
  • 選項 B：扣除台度太苛刻 → 法規這樣寫的，沒得改
  • 選項 C：把房間定義改成「儲藏室」就不用查採光 → 業主想要當臥室
  • 選項 D：跳出 AI，找老闆/業主討論（P12 死局）

為什麼 MCP 不該幫你選

每個選項的權重涉及業主關係、預算、立面美學、結構可行性、未來改建空間⋯⋯這些 MCP 不知道。把光譜決策交給 AI 等於把「設計判斷」這個職業核心交出去。

死局時跳出找人

P12

使用者原文

一旦被限制住，我認為人應該要跳脫與人機互動的死局，反而應該要去求助更多的【人】來進行腦力激盪。 — 使用者，2026-05-18

死局判定

當你發現自己在問 AI 同一個問題的不同變形、AI 一直給類似建議都沒解、感覺被困住——那就是死局。

正確做法

關掉 AI 對話。打電話給老闆、業主、結構技師、室內設計師、之前合作過的同事。讓「人 + 人」的腦力激盪上場，因為這個層級的問題本來就是 AI 解不了的。

AI 認輸是好事

一個會說「這超出我能解的範圍」的 AI，比一個會硬掰建議的 AI 好得多——前者誠實、後者危險。

  • 不主動給「該怎麼修」（對應 P13）

5/22 dry-run 新發現

  • Tool Scope Mismatch（L-026）：check_exterior_wall_openings 不吃 level 參數，回的 8 項違規跨多個樓層（z 座標分布在 0/100/3170）。AI 應主動 surface「整案結果跨樓層，請勿假設都在當前樓層」

報告解讀 + 失敗 fallback

三條決策路徑

  1. 全 PASS（GO）→ 標記 LOD 300 達成，進下階段

  • 某個 check_* 工具 timeout → 跳過該工具，繼續其他

  • 連結模型問題（P22 ARCHI 範圍內不發生）
