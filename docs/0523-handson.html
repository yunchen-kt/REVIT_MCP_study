核心練習：步驟 0-6

每個藍色 code block 都是「直接貼到 AI Client 對話框」的 prompt。執行後對照「預期觀察」確認。卡住請看下方常見問題。
設計原則：每個 prompt 單一意圖、語意完整，避免 LLM 自由發揮。對應命題 P8「被動就緒」——你不問，AI 不答。

CORE STEP 0 / 6

Anchor — 讀使用者當下實際在哪

為什麼這步：對應命題 P8「被動就緒」+ 憲法級「Tool Call Data Honesty」+ 「Active State Re-Anchoring」（5/22 新增）。先讓 AI 讀使用者當下實際所在的視圖/樓層，**所有後續操作以這個 anchor 為基準**——不假設使用者在 1F，不依賴 handson 文件預設的「日本 sample 應該是 53189012」這類 hardcoded 期望。

COPY 請呼叫 get_active_view 告訴我我現在在 Revit 哪個視圖。 請列出：視圖名稱、類型、樓層、ViewId（4 項都要）。 不要做任何進一步推論。
✓ 預期觀察
  • AI 回一行：「您目前在『{視圖名}』，類型『{FloorPlan/Section/...}』，樓層『{LevelName}』，ViewId={數字}。」
  • PASS 如果 AI 開始建議「您可以做 X、Y、Z」——那是它違規（沒被問就答）。命題 P8 被破壞的樣子
  • FAIL AI 列出具體房間 ID 但你沒呼叫 get_rooms_by_level——違反 Tool Call Data Honesty
  • 📌 把 ViewId 與 LevelName 抄下來，但**這只是「Step 0 時的 anchor」**——後續每個 level/view-anchored 步驟（Step 2/3/5）AI 都應該重新 re-anchor，不依賴 Step 0 的 snapshot
📌 Active State Re-Anchoring 規則（5/22 新增，憲法級）

**任何引用 view-state / level-state 的 claim 之前，AI 必須在 claim 時點重新呼叫 get_active_view 確認**——不能依賴 session 較早的 read 結果。使用者隨時可能切視圖（這是真實設計師工作流，一邊看作業用、一邊翻法規、一邊跳到矩計），AI 沒有任何「異常」可偵測，只是在 stale snapshot 上繼續做 claim。詳見 domain/session-context-guard.md 與 domain/lessons.md L-025。

CORE STEP 1 / 6

確認當下視圖是否適合後續操作（read-only intake）

為什麼這步：5/22 redesign。原版設計「強制切到 1F」是錯的——真實設計師不會配合 demo 文件預期，他們會繼續按自己工作流切視圖。Step 1 改為 read-only intake：確認 Step 0 抄下的視圖類型 / 樓層**適不適合**後續 ARCHI 合規檢查；若不適合（例如使用者在 3D 視圖 / 剖面視圖 / 非檢討樓層），由使用者決定要不要手動切到合適的平面。

COPY 根據 Step 0 抄下的視圖類型與樓層，請判斷後續 ARCHI 合規檢查（外牆開口、排煙、樓梯淨高、採光、樓板有效開口）是否能在這個視圖上正常進行： - 視圖類型必須是 FloorPlan（平面）——若是 3D/Section/Elevation，請告訴我「請手動切到平面視圖再繼續」 - 樓層必須是有實際房間定義的（不能是 GL/設計GL/外構等）——若是無效樓層，請告訴我「此樓層無房間資料，請手動切到 1F-7F 任一」 - 若視圖名稱含「詳図/作業用/法規チェック/階段/建具/矩計/展開」等專用前綴，視圖內容可能被 view template 過濾——建議切到「平面図 X 階」這類主平面視圖 不要呼叫 set_active_view 替我切視圖——讓我自己決定切到哪。 回報格式：「您目前的視圖『{name}』是 {OK/不建議}，原因：{...}。」
✓ 預期觀察
  • AI **不呼叫 set_active_view**——這是 read-only intake，光譜決策（要不要切視圖）在使用者端
  • AI 給出 OK / 不建議的判斷 + 一句原因
  • 若視圖 OK → 直接進 Step 2，**Step 2 會以這個視圖的 LevelName 為基準**
  • 若視圖不建議 → 你手動切到合適平面（或不切，後續結果可能異常）
  • PASS AI 沒幫你決定要切哪個視圖——光譜在使用者端
  • FAIL AI 自己 set_active_view 切到「它認為對的」視圖——這違反 Step 1 的 read-only intake 精神
⚠ 真實工作流場景（5/22 dry-run 實證）

5/22 dry-run 中段使用者刻意在 Step 1 後再切 6F → 2F duplicate → 不告訴 AI。**正確的 AI 應該在每個後續步驟（Step 2/3/5）開頭都重 re-anchor**，跟著使用者眼前的畫面跑，而不是綁死在 Step 0/1 抄下的 LevelName。這對應雙向 Context Sync 協議——使用者切視圖是天然行為，AI 應主動跟上。

CORE STEP 2 / 6

看看這層樓有什麼

為什麼這步：對應命題 P7「階段完成點」——我們先看「具象成果」是什麼，再決定要檢什麼。

COPY 請呼叫 get_rooms_by_level 帶 level="<把 Step 0/1 抄下的 LevelName 填這，例如日本 sample 是 1FL>"。 列出這層樓所有房間：ElementId、Number、Name、Area。 表格呈現，不要加任何分析建議。
✓ 預期觀察
  • AI 回一張 markdown 表格，欄位：Room Number / Name / Area
  • 房間數量視日本 sample 內容而定（住宅 sample 約 5-10 間）
  • PASS AI 沒主動分析「您的客廳偏小」——這是它正確遵守「不問不答」
💭 哲學意義

這個列表就是命題 P7「具象成果」——你看著它，腦袋才有東西可以跑光譜決策。如果沒列出，後面所有檢查都是抽象的。

觸發 ARCHI 合規批次檢查

為什麼這步：這是整場 Live Demo 的核心。對應命題 P4「限制顯現器」+ P16「Work Breakdown 讓階段完成點可識別」+ P22「ARCHI 優先」。一個 prompt 編排 5 個既有工具並行檢查，這就是「91 工具編排平台 = 業界真空白」的具體呈現。

✓ 預期觀察（已對齊 2026-05-18 修復後實機）
  • AI 並行（或快速序列）呼叫 5 個工具（含 check_stair_headroom levelName 批次模式）
  • 5 個工具全部正常回應，無 C# crash、無 output 超限

  • PASS AI 沒幫你寫「建議改為 1500mm 寬」——對應 P13「MCP 是 0/1 執行器」

  • PASS AI 對 NeedsManualConfirm 的回應是誠實標示（如「無法判定開啟方式，需人工確認」），不是編造數據——對應 Tool Call Data Honesty

⚠ Tool Scope Mismatch（5/22 新增）— 同批 5 工具的回應範圍不一致

這 5 個工具中，1 個是 project-wide、4 個是 level-scoped：

  • check_exterior_wall_openings → project-wide（不吃 level 參數，掃 445 牆全跑）
  • check_smoke_exhaust_windows / check_stair_headroom / get_room_daylight_info / check_floor_effective_openings → level-scoped（吃 levelName / level 參數）

**這意味著**：若你在 2FL 跑這 5 工具，check_exterior_wall_openings 回的 8 項違規 **跨越多個樓層**（含 1F 開口 z=0/100 + 2F 高窗 z=3170），但被混在「2FL 合規報告」中容易誤判。AI 應主動 surface：「以下 5 個工具中，4 個是 X 樓層範圍、1 個是整案範圍——整案結果跨越多樓層，請勿假設它們都發生在當前樓層」。詳見 domain/tool-capability-boundary.md L7 與 domain/lessons.md L-026。

CORE STEP 4 / 6

對某個 FAIL 深入查（光譜決策的起點）

為什麼這步：對應命題 P11「光譜在設計師端」+ P10「設計 = 解決問題，美學 = 手段」。AI 告訴你「FAIL」，接下來怎麼想是你的事，不是 AI 的事。

把 {ROOM_ID} 換成你想深入的房間 ElementId（如 52842360＝駐車場 27 是 5/18 實機跑出的排煙 FAIL 房間，剛好做雙重交叉驗證）
COPY 請呼叫 get_room_daylight_info 帶 level=1FL 取得本層所有房間採光資料。 從報告中找出 ROOM_ID = {ROOM_ID} 那個房間，列出： - 房間面積（m²） - 該房間所有開口的：Category / Width / Height / SillHeight / HeadHeight / IsExterior - 依 domain/daylight-area-check.md 公式逐筆計算： · 若 SillHeight ≥ 750mm：Effective Area = Width × Height · 若 SillHeight < 750mm：Effective Height = HeadHeight - 750mm，Effective Area = Width × Effective Height（負值取 0） · 僅納入 IsExterior=true 的窗 + IsExterior=true 且含玻璃的門 - 合計 Σ Effective Area - 採光比 = Σ Effective Area / 房間面積 不要建議任何修改方案，純報告數據。
✓ 預期觀察（已對齊實機）
  • AI 呼叫 get_room_daylight_info 帶 level=1FL 參數（工具不接 roomId——schema 只有 level 一個 optional 參數）
  • 回 raw data：本層 20 個房間 + 每房間 Openings 含 SillHeight / HeadHeight / IsExterior / Category
  • AI 自行套 75cm 公式算 Effective Height（工具不算）→ 算採光比 = 小數（如 0.087 = 8.7%）
  • PASS AI 應該引用 domain/daylight-area-check.md 的公式——這是「Domain Method Compliance」憲法生效
  • FAIL 如果 AI 直接用工具回的「數字」當採光面積，沒套 75cm 扣除——它違規 Domain Method Compliance
⚠ 這步是「兩條憲法同時生效」的試金石
  • Tool Call Data Honesty：房間面積、SillHeight 等數據必須來自 tool response，AI 不能編
  • Domain Method Compliance：採光比的計算公式必須來自 domain/daylight-area-check.md，AI 不能用「常識」算
  • 看 AI 怎麼把 raw data 跟 domain SOP 接起來——這是 0523 哲學體系最核心的演示時刻
💭 哲學意義 — 光譜決策的起點（適用任何工具 FAIL，不限採光）

不論你拿到的 FAIL 是採光、是排煙、是樓梯淨高、是外牆開口——站在 FAIL 結果前的光譜決策結構是一樣的：

選項 A：直接修方案（加窗 / 加排煙設備 / 改樓梯）→ 結構衝突？立面美感衝突？預算？
選項 B：發現規範的扣除/限制太嚴苛 → 法規這樣寫，沒得改，認了
選項 C：改變空間定義（居室→儲藏室 / 一般空間→非檢討範圍）→ 業主的真實需求允許嗎？
選項 D：跳出 AI，找老闆 / 業主 / 結構技師 / 室內設計師（命題 P12 死局求助更多人）

MCP 把這個選擇還給了你——它不替你決定，但它讓你知道你在決定什麼。

5/18 日本 sample 實測：真正回 FAIL 的是排煙（店舗 26 / 駐車場 27 / ピロティ 141 三間無窗居室）。同樣的 A/B/C/D 框架直接套——A 加排煙窗影響立面 / B 規範就 2% 沒得議價 / C 把店舗改非居室業主不肯 / D 跳出找業主談機械排煙預算。看吧，框架普適。

CORE STEP 5 / 6

視覺化「規範限制施加的位置」（依規範類型分支：wall-anchored 直染 / room-anchored 走 proxy）

為什麼這步：對應 slide 6-4「報告解讀 + 決策路徑」三條路徑命題（P13 MCP 給 0/1、P11 光譜在設計師端、P12 死局求助）+ P3「限制可見」+ P4「限制顯現器」。視覺化的對象是「規範在哪裡施加限制」（規範本體），不是「AI 判定為 FAIL 的房間」（判定結果）。後者違反 P5/P6 人機分工翻轉——把 AI 判定結果包裝太顯眼就是搶走設計師的光譜決策。
