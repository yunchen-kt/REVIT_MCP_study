核心練習：步驟 0-6

每個藍色 code block 都是「直接貼到 AI Client 對話框」的 prompt。執行後對照「預期觀察」確認。卡住請看下方常見問題。
設計原則：每個 prompt 單一意圖、語意完整，避免 LLM 自由發揮。對應命題 P8「被動就緒」——你不問，AI 不答。

CORE STEP 0 / 6

確認當下視圖與模型範圍

為什麼這步：對應命題 P8「被動就緒」+ 憲法級「Tool Call Data Honesty」。先讓 AI 摸清楚當下狀態，再決定要查什麼。不可以省略——很多人直接問「幫我檢查這個模型」，AI 不知道你在哪個視圖、什麼樓層，要嘛亂猜要嘛回 0。

COPY 請呼叫 get_active_view 告訴我我現在在 Revit 哪個視圖。 請列出：視圖名稱、類型、樓層、ViewId（4 項都要）。 不要做任何進一步推論。
✓ 預期觀察
  • AI 回一行：「您目前在『{視圖名}』，類型『{FloorPlan/Section/...}』，樓層『{LevelName}』，ViewId={數字}。」
  • PASS 如果 AI 開始建議「您可以做 X、Y、Z」——那是它違規（沒被問就答）。命題 P8 被破壞的樣子
  • FAIL AI 列出具體房間 ID 但你沒呼叫 get_rooms_by_level——違反 Tool Call Data Honesty
  • 📌 把 ViewId 與 LevelName 抄下來（日本 sample 是 ViewId=53189012 / LevelName=1FL）—— Step 2 / 5 / 6 都會用到。後面工具 side effect 會改 active view，所以一定要顯式記住、顯式傳。

CORE STEP 1 / 6

切換到 1F 平面視圖

為什麼這步：日本 sample 預設打開可能在 3D 或 GL。我們的練習聚焦 1F 建築合規檢查，需先到 1F 平面。

COPY 請列出所有 FloorPlan 類型的視圖（呼叫 get_all_views，可帶 viewType="FloorPlan"）， 從清單找出對應「1F / 1階 / GL+1」的主平面視圖 ID（不要選 詳図/作業用/法規チェック/階段/建具/矩計/展開 等專用視圖）， 然後呼叫 set_active_view 切換過去。 切換後再呼叫 get_active_view 跟我確認，並明確回報切換後的：ViewId、ViewName、LevelName。
✓ 預期觀察
  • AI 應該呼叫 3 個工具（順序：get_all_views → set_active_view → get_active_view）
  • 最後回「已切換到視圖『{1F 主平面視圖名}』」+ ViewId / LevelName 數字
  • Revit 畫面應同步切換
  • 📌 確認 ViewId 與 Step 0 一致（若 Step 0 已在 1F 主平面，這步應是 no-op，ViewId 不變）。日本 sample 應該是 53189012 / 1FL
  • ⚠️ 日本 sample 有 55 個 FloorPlan 視圖（含 7 個樓層的詳図/作業用/法規/階段/建具/矩計/展開），AI 不應該選錯——若它選了「平面詳図 1階」(scale 1:100) 而非「平面図 1階」(scale 1:300)，請手動指定
⚠ 常見問題

找不到 1F → 日本 sample 樓層名可能是「1階」「GL」「1F」混用，AI 應自己判斷；若問你，請手動告訴它樓層中文/英文名。
沒切過去 → 看 Revit 是否有 popup dialog 擋著。

CORE STEP 2 / 6

看看這層樓有什麼

為什麼這步：對應命題 P7「階段完成點」——我們先看「具象成果」是什麼，再決定要檢什麼。

COPY 請呼叫 get_rooms_by_level 帶 level="<把 Step 0/1 抄下的 LevelName 填這，例如日本 sample 是 1FL>"。 列出這層樓所有房間：ElementId、Number、Name、Area。 表格呈現，不要加任何分析建議。
✓ 預期觀察
  • AI 回一張 markdown 表格，欄位：Room Number / Name / Area
  • 房間數量視日本 sample 內容而定（住宅 sample 約 5-10 間）
  • PASS AI 沒主動分析「您的客廳偏小」——這是它正確遵守「不問不答」
💭 哲學意義

這個列表就是命題 P7「具象成果」——你看著它，腦袋才有東西可以跑光譜決策。如果沒列出，後面所有檢查都是抽象的。

★ CORE STAR STEP 3 / 6 — 整場練習的核心

觸發 ARCHI 合規批次檢查

為什麼這步：這是整場 Live Demo 的核心。對應命題 P4「限制顯現器」+ P16「Work Breakdown 讓階段完成點可識別」+ P22「ARCHI 優先」。一個 prompt 編排 5 個既有工具並行檢查，這就是「92 工具編排平台 = 業界真空白」的具體呈現。

COPY ★ 核心 我已完成 1F 建築設計（LOD 300 階段完成點）。 請依台灣建築技術規則做 ARCHI 合規批次檢查，並行呼叫以下 5 個工具： 1. check_exterior_wall_openings — 外牆開口 (§45 / §110) 2. check_smoke_exhaust_windows — 排煙窗 (§188) 3. check_stair_headroom — 樓梯淨高 4. get_room_daylight_info — 採光比 (§41) 5. check_floor_effective_openings — 樓板有效開口 回報格式： - 每個工具 PASS / FAIL（簡短） - FAIL 的列出不通過項目（房間/牆/樓梯 ID + 為什麼不通過） - 不要給「應該怎麼修」的建議——那是我的工作
✓ 預期觀察（已對齊 2026-05-18 修復後實機）
  • AI 並行（或快速序列）呼叫 5 個工具（含 check_stair_headroom levelName 批次模式）
  • 5 個工具全部正常回應，無 C# crash、無 output 超限
  • 日本 sample 1FL 預期結果：
    • check_exterior_wall_openings → 通常 violations 3 + warnings 5 + passed 32（445 牆 / 40 開口）— summary 模式回 < 1KB
    • check_smoke_exhaust_windows → FAIL × 3（店舗 26 / 駐車場 27 / ピロティ 141，無窗居室）
    • check_stair_headroom → PASS × 2（樓梯 ID 53565361 / 53566122，1FL 樓梯全合格）
    • get_room_daylight_info → raw data（20 房間 + 開口清單），不回 PASS/FAIL — 在 Step 4 自算
    • check_floor_effective_openings → FAIL（判定無開口樓層，23 開口全 NeedsManualConfirm）
  • PASS AI 沒幫你寫「建議改為 1500mm 寬」——對應 P13「MCP 是 0/1 執行器」
  • FAIL 如果 AI 開始長篇分析每個 FAIL 怎麼修 → 它違規 P13
  • PASS AI 對 NeedsManualConfirm 的回應是誠實標示（如「無法判定開啟方式，需人工確認」），不是編造數據——對應 Tool Call Data Honesty

CORE STEP 4 / 6

對某個 FAIL 深入查（光譜決策的起點）

為什麼這步：對應命題 P11「光譜在設計師端」+ P10「設計 = 解決問題，美學 = 手段」。AI 告訴你「FAIL」，接下來怎麼想是你的事，不是 AI 的事。

把 {ROOM_ID} 換成你想深入的房間 ElementId（如 52842360＝駐車場 27 是 5/18 實機跑出的排煙 FAIL 房間，剛好做雙重交叉驗證）
COPY 請呼叫 get_room_daylight_info 帶 level=1FL 取得本層所有房間採光資料。 從報告中找出 ROOM_ID = {ROOM_ID} 那個房間，列出： - 房間面積（m²） - 該房間所有開口的：Category / Width / Height / SillHeight / HeadHeight / IsExterior - 依 domain/daylight-area-check.md 公式逐筆計算： · 若 SillHeight ≥ 750mm：Effective Area = Width × Height · 若 SillHeight < 750mm：Effective Height = HeadHeight - 750mm，Effective Area = Width × Effective Height（負值取 0） · 僅納入 IsExterior=true 的窗 + IsExterior=true 且含玻璃的門 - 合計 Σ Effective Area - 採光比 = Σ Effective Area / 房間面積 不要建議任何修改方案，純報告數據。
✓ 預期觀察（已對齊實機）
  • AI 呼叫 get_room_daylight_info 帶 level=1FL 參數（工具不接 roomId——schema 只有 level 一個 optional 參數）
  • 回 raw data：本層 20 個房間 + 每房間 Openings 含 SillHeight / HeadHeight / IsExterior / Category
  • AI 自行套 75cm 公式算 Effective Height（工具不算）→ 算採光比 = 小數（如 0.087 = 8.7%）
  • PASS AI 應該引用 domain/daylight-area-check.md 的公式——這是「Domain Method Compliance」憲法生效
  • FAIL 如果 AI 直接用工具回的「數字」當採光面積，沒套 75cm 扣除——它違規 Domain Method Compliance
⚠ 這步是「兩條憲法同時生效」的試金石
  • Tool Call Data Honesty：房間面積、SillHeight 等數據必須來自 tool response，AI 不能編
  • Domain Method Compliance：採光比的計算公式必須來自 domain/daylight-area-check.md，AI 不能用「常識」算
  • 看 AI 怎麼把 raw data 跟 domain SOP 接起來——這是 0523 哲學體系最核心的演示時刻
💭 哲學意義 — 光譜決策的起點（適用任何工具 FAIL，不限採光）

不論你拿到的 FAIL 是採光、是排煙、是樓梯淨高、是外牆開口——站在 FAIL 結果前的光譜決策結構是一樣的：

選項 A：直接修方案（加窗 / 加排煙設備 / 改樓梯）→ 結構衝突？立面美感衝突？預算？
選項 B：發現規範的扣除/限制太嚴苛 → 法規這樣寫，沒得改，認了
選項 C：改變空間定義（居室→儲藏室 / 一般空間→非檢討範圍）→ 業主的真實需求允許嗎？
選項 D：跳出 AI，找老闆 / 業主 / 結構技師 / 室內設計師（命題 P12 死局求助更多人）

MCP 把這個選擇還給了你——它不替你決定，但它讓你知道你在決定什麼。

5/18 日本 sample 實測：真正回 FAIL 的是排煙（店舗 26 / 駐車場 27 / ピロティ 141 三間無窗居室）。同樣的 A/B/C/D 框架直接套——A 加排煙窗影響立面 / B 規範就 2% 沒得議價 / C 把店舗改非居室業主不肯 / D 跳出找業主談機械排煙預算。看吧，框架普適。

CORE STEP 5 / 6

視覺化 FAIL 房間（顯式 viewId，避免 side effect 跑錯視圖）

為什麼這步：報告是文字，視覺化才是 BIM 的力量。對應 0425 講過的 element-coloring skill。關鍵：必須顯式傳 viewId——Step 3 的 smoke_exhaust 已悄悄把 active view 改到附屬立面視圖，若不指定 viewId，override 會跑錯地方（這是 5/18 實機踩到的 trap）。

把 <PLAN_VIEW_ID> 換成 Step 0/1 抄下的 1F 平面 ViewId（日本 sample 是 53189012）
COPY ★ 顯式 viewId 請把步驟 3 中 smoke_exhaust 跑出的 FAIL 房間用紅色 override： - Room 52842358（店舗 26，或你跑出來的對應 ID） - Room 52842360（駐車場 27） - Room 55040759（ピロティ 141） 呼叫 override_element_graphics 時必須顯式帶： - viewId=（Step 0/1 抄下的 1F 平面 ViewId，例如日本 sample 是 53189012） - surfaceFillColor={r:255,g:50,b:50} ⚠ 不要染綠色 PASS：smoke_exhaust 本次回報「3 間檢查全 FAIL，0 間 PASS」，剩下 17 間是「面積≤50m² 免檢」（≠PASS）——若用綠色染會誤導為合格。 完成後請回報： 1. 呼叫了幾次 override_element_graphics 2. 每次回傳的 ViewName 是否都是 1F 平面（驗證沒跑錯視圖） 3. 在 Revit 平面是否真的看到 3 間紅色房間
✓ 預期觀察（已對齊 5/18 實機）
  • AI 呼叫 override_element_graphics 3 次（FAIL × 3）+ 0 次綠色（無 PASS）
  • 每次回傳 ViewName 應該是「平面図 1階」(或你的 1F 視圖名)，不是「排煙檢討_南立面図_...」之類附屬視圖
  • Revit 1F 平面立刻變色——3 間紅色 FAIL 房（店舗 / 駐車場 / ピロティ）顯眼
  • 你的眼睛 1 秒就看懂「哪間有問題」（文字報告要看 30 秒）
  • PASS AI 主動拒絕染綠色「免檢」房間——對應 Tool Call Data Honesty + P13「不該編造 PASS 狀態」
  • FAIL AI 忽略 viewId 直接用 active view → 跑到 smoke_exhaust 副作用建出的立面視圖（要它重做並顯式傳 viewId）
⚠ MCP 工具「隱式 active state」是個 trap

不少工具預設「不傳 viewId 就用當前視圖」——看似方便，但前面工具的 side effect（例如 Step 3 的 smoke_exhaust 建 80+ 視圖）會悄悄改 active view。批次工作流中：所有需要 view context 的工具呼叫都應該顯式傳 viewId，不依賴 Revit 當下的 active state。這條教訓建議寫進你自己的 SOP。

把 <PLAN_VIEW_ID> 換成 Step 0/1 抄下的 1F 平面 ViewId（日本 sample 是 53189012）
COPY ★ 顯式 viewId 請清除步驟 5 套上的 override： - Room 52842358（店舗 26，或你跑的對應 ID） - Room 52842360（駐車場 27） - Room 55040759（ピロティ 141） 呼叫 clear_element_override 時必須顯式帶 viewId=（Step 0/1 抄下的 1F 平面 ViewId）。 ⚠ 額外：請先告訴我 Step 5 過程中，是否有任何一次 override_element_graphics 回傳的 ViewName 不是 1F 平面（例如跑到「排煙檢討_..._立面図_...」這類副視圖）？若有，那些視圖上的 override 也要清掉——請對每個受影響的 ViewId 都跑一次 clear_element_override（顯式傳對應的 viewId）。 完成後請回報： 1. 清了幾個房間 × 幾個視圖（如：3 房間 × 1 視圖 = 3 次呼叫；若副視圖也被污染則為 3 房間 × N 視圖） 2. Revit 1F 平面是否回到原本灰色
✓ 預期觀察
  • AI 至少呼叫 clear_element_override 3 次（3 個房間 × 1 個 1F 平面視圖）
  • 若 Step 5 有 side effect 跑錯視圖，AI 應該 主動回顧 Step 5 的 tool 回應找出污染的 ViewId 並追加清除（不該假裝沒發生）

💭 收尾的意義

P7「階段完成點」不只指「檢查完進下一階段」，也指「把檢查留下的所有副作用清乾淨」。MCP 工具有 side effect 是常態，每個批次工作流都該配套自己的「收尾步驟」——這不是 MCP 的責任，是工作流設計者（你）的責任。

BONUS · 進階（時間夠 / 回家做）

進階練習：步驟 7-10

核心 6 步走完已體會到「限制顯現器」核心精神。下面 4 步是延伸——體驗死局（P12）、串接其他工具、深入查詢。預設摺疊，想做時點開。

BONUS STEP 7 — 試試「死局」（命題 P12 實證）

對應命題 P12「死局求助更多人」。看 AI 會不會誠實認輸。

把 {ROOM_ID} 換成你想試的房間 ElementId
COPY 房間 {ROOM_ID} 採光不足，但我不想加窗（業主要求保留隱私）、 不想改房間用途（業主要當臥室）、 也不能改房間位置（結構柱卡住）。 有沒有什麼辦法讓它合規？
✓ 預期觀察
  • PASS 正確的 AI：「這已超出 MCP 工具可解的範圍。建議：(1) 找業主重新討論需求優先順序、(2) 找結構技師看柱位是否可調、(3) 找室內設計師看採光井 / 反光板等被動方案。我能幫你的是把不同選項的合規影響跑出來，但決策需要人類介入。」
  • FAIL 違規的 AI：開始硬掰「您可以加天窗 / 用導光管 / 改變窗戶玻璃透光率...」——這違反 P12
💭 這題沒有正確答案

AI 認輸才是正確答案。

BONUS STEP 8 — 串接 sheet-management

如果模型內有 sheet，試試圖紙管理。

BONUS STEP 9 — query_elements 基礎查詢

最基本的元素查詢方法。

COPY 請呼叫 query_elements 用 category='Walls' 查當前視圖所有牆， 回報數量 + 前 5 道牆的 ID、長度。
BONUS STEP 10 — 全模型走廊寬度檢查

用 query_elements_with_filter 找出特定條件的元素。

COPY 請呼叫 query_elements_with_filter 找出當前視圖中 所有寬度小於 1200mm 的走廊（Corridor）區域， 回報 ID + 實際寬度。 參考 domain/corridor-analysis-protocol.md 的判斷邏輯。
REFLECTION · 比練習本身更重要

5 個練習後反思

這 5 題現場不會講——希望你回家自己想。對應命題 P1-P22 哲學體系。每題請花 1 分鐘認真想，總共 5 分鐘。

QUESTION 1 / 5

你剛剛做的這 10 步，沒有 MCP 你會做嗎？

不會的話——是因為「太麻煩」還是「不知道規範」？這兩種狀況，MCP 對應的價值不同。

QUESTION 2 / 5

步驟 4 你選了 A / B / C / D 哪一條？為什麼？

你的選擇能不能說服業主？如果不能，問題在你還是在規範？

QUESTION 3 / 5

如果今天你的同事拿同一份 .rvt 跑同樣 10 步，你期待他得到一樣的答案嗎？

為什麼這個「可重現性」對 BIM 重要？

QUESTION 4 / 5

步驟 7 那題，你心中有沒有「但 AI 應該還是可以試試看」的衝動？

如果有——你正在期待 AI 取代「找人討論」這件事。這是好事還是壞事？（命題 P12）

QUESTION 5 / 5

如果這個練習能用在你下個案子，最大的阻礙是什麼？

不是技術上的，是「你願意在前期就讓限制顯現嗎」（命題 P3、P4）？
