Best Practices

📜 CLAUDE.md 最佳實踐

  • 控制在 < 200 行，超過後遵從率下降
  • 用 MUST / NEVER 強制語氣（比 "Prefer X" 有效 3 倍）
  • 結構：WHAT（技術棧）→ WHY（目的）→ HOW（工作方式）
  • 每條規則自問：移除會讓 Claude 犯錯嗎？不會就刪
  • 分層放置：根目錄 + 子目錄各自 CLAUDE.md
  • 用 @path/to/file 引入額外檔案，避免主檔過長
  • 用 /init 自動生成初版，再手動精修
  • commit 進 git，團隊共維，月度 review 更新
"Claude 做錯就加進 CLAUDE.md" — Boris

📈 Context 管理策略

  • 在 70-80% 時手動 /compact（比自動壓縮品質好）
  • Subagent 探索：主 context 只收 ~2K 摘要（省 93%）
  • 停用不用的 MCP（幾個就能吃掉 30%+ 窗口）
  • 給精確檔案路徑比模糊描述省 10 倍 context
  • 30-45 分鐘 session > 馬拉松 session
  • 用 /context 指令審計 token 消耗來源
"想投入更多算力就在 prompt 加 'use subagents'" — Boris

⚡ Workflow 高手技

Plan → Auto-accept 模式

  • Plan mode 反覆討論到滿意 → 切 auto-accept 一次完成
  • Ctrl+G 在外部編輯器中直接編輯 plan

平行 Session 策略

  • 終端 5 個 + browser 5-10 個 session 同時跑
  • 每個 session 一個 branch（絕不共用 branch）
  • 開 3-5 個 git worktree，tab 編號 1-5 方便切換
claude --worktree feature-x --tmux
claude -w  # 自動命名 worktree

三層開發架構

層級說明
Tier 1互動式開發（單一 session）
Tier 2平行衝刺（Agent Teams / worktree）
Tier 3整夜跑批次（headless + CI/CD）
"給 Claude 驗證方法，品質提升 2-3 倍" — Boris

Tips & Resources

🐛 Debug 高效流程

  • Director Mode：給目標（不是給步驟）
  • 貼完整錯誤訊息 + 檔案路徑（不要只說「出錯了」）
  • 卡在 debug 迴圈時用 think harder 或 ultrathink
  • 讓 Claude 自己讀日誌、跑測試、追 stack trace
  • 在 CLAUDE.md 加常用 debug 指令

🔍 Code Review 進階

  • 安裝 GitHub integration 自動審查 PR
  • 用 REVIEW.md 或 CLAUDE.md 自訂審查規則
  • 大 PR (1000+ 行)：84% 被標記問題，平均 7.5 個發現
  • 好的 review prompt 要定義 scope + risk areas

自訂指令支援位置引數 $1 $2 和 $ARGUMENTS

---
allowed-tools: Read, Grep, Glob
---
分析 $ARGUMENTS 的程式碼品質

🛡 安全與權限設定

  • deny > ask > allow（deny 永遠優先）
  • defaultMode: default | acceptEdits | plan | bypassPermissions
  • additionalDirectories 加入額外可存取目錄
  • 團隊用 managed-settings.json 強制統一安全策略
