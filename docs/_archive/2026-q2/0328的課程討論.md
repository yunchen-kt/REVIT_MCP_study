## 1-1. MCP 是什麼（2 分鐘）

### 逐字稿

> 我們先花兩分鐘快速複習 MCP 的基本概念，如果你是第一次來也不用擔心，這段會讓你抓到重點。
>
> MCP 全名是 Model Context Protocol，它是一個開放標準，讓 AI 應用程式可以去「操作」外部的工具。
>
> 你可以想像成這樣：平常我們用 ChatGPT 或 Claude 聊天，它只能「講話」，不能真的幫你做事。但有了 MCP，AI 就像長出了手，可以直接去操作 Revit。
>
> 整個流程是這樣的：

**（投影片顯示架構圖）**

```
你（使用者）
  ↓ 用自然語言講話，例如「幫我查 3 樓所有房間的面積」
AI Client（你用的 AI 軟體：Claude Desktop / Gemini CLI / VS Code）
  ↓ AI 判斷要呼叫哪個工具，透過 MCP 協議傳遞
MCP Server（Node.js 程式，跑在你的電腦上）
  ↓ 透過 WebSocket 連線（預設 port 8964）傳給 Revit
Revit Add-in（C# 外掛，裝在你的 Revit 裡面）
  ↓ 呼叫 Revit API 實際執行操作
Revit 執行完畢 → 結果回傳 → AI 用自然語言告訴你結果
```

> 所以整個鏈條有三個東西要裝：
>
> 1. **AI Client** — 你跟 AI 講話的地方（Claude Desktop、Gemini CLI、VS Code 都可以）
> 2. **MCP Server** — 翻譯官，把 AI 的請求翻譯成 Revit 聽得懂的格式
> 3. **Revit Add-in** — 真正在 Revit 裡面動手的外掛
>
> 這三個東西都跑在你自己的電腦上，不會把你的模型傳到雲端。

### 講者備註
- 如果現場有第一次來的人，停下來問「到這裡有問題嗎？」
- 強調「你的模型不會上傳」，這是實務上很多人會擔心的

## 1-2. 三個東西各自裝在哪裡（3 分鐘）

### 逐字稿

> 接下來我們看這三個東西分別放在電腦的哪裡。這個很重要，因為很多部署問題都是路徑搞錯。

**（投影片顯示路徑表格）**

> **第一個：MCP Server（Node.js）**
>
> 它就是我們 GitHub 下載下來的專案裡面的 `MCP-Server` 資料夾。你 clone 專案放在哪裡，它就在哪裡。
>
> ```
> <你 clone 的位置>\REVIT_MCP_study\MCP-Server\
>   ├── src\         ← TypeScript 原始碼
>   ├── build\       ← 編譯後的 JavaScript（AI Client 指向這裡）
>   └── package.json
> ```
>
> 你需要跑 `npm install` 和 `npm run build` 來編譯它。

> **第二個：Revit Add-in（C# DLL）**
>
> 它要部署到 Revit 專屬的 Addins 資料夾。這個路徑是固定的：
>
> ```
> C:\Users\你的名字\AppData\Roaming\Autodesk\Revit\Addins\2024\
>   ├── RevitMCP.addin          ← 告訴 Revit「有這個外掛」
>   └── RevitMCP\
>       └── RevitMCP.dll        ← 外掛本體
> ```
>
> 注意兩件事：
> - `AppData` 是隱藏資料夾，你要在檔案總管打 `%APPDATA%` 才看得到
> - **只能有一個 .addin 檔案**。如果你看到 `RevitMCP.2024.addin`，那是舊版殘留，要刪掉，否則 Revit 會載入兩次

> **第三個：AI Client 的設定檔**
>
> 每個 AI Client 有自己的設定檔位置，裡面要告訴它「MCP Server 在哪裡」：

| AI Client | 設定檔位置 | 備註 |
|-----------|-----------|------|
| Claude Desktop | `%APPDATA%\Claude\claude_desktop_config.json` | 改完要重開 Claude |
| Gemini CLI | `~/.gemini/settings.json`（Windows: `%USERPROFILE%\.gemini\settings.json`） | 不用重開 |
| VS Code Copilot | 專案根目錄的 `.vscode/mcp.json` | 可以用 `${workspaceFolder}` 變數 |

> 設定檔的內容長這樣，三個 Client 格式都一樣：
>
> ```json
> {
>   "mcpServers": {
>     "revit-mcp": {
>       "command": "node",
>       "args": ["<你的完整路徑>/REVIT_MCP_study/MCP-Server/build/index.js"]
>     }
>   }
> }
> ```
>
> **⚠️ 請把 `<你的完整路徑>` 換成你實際 clone 的位置**，例如 `C:/Users/你的名字/Desktop`。用正斜線 `/`，不要用反斜線 `\`。路徑錯誤是最常見的連線失敗原因。

### 講者備註
- 建議投影片上放一張截圖：Windows 檔案總管打開 `%APPDATA%` 的畫面
- 強調 `.addin` 只能一個，這是上次幾個人踩到的坑

## 2-1. 問題一：工具越多，AI 越笨（5 分鐘）

### 逐字稿

> 好，進入今天的第一個重點。我們先不講解法，先講「我們遇到了什麼問題」。因為如果你不知道問題在哪，等一下聽解法會覺得「幹嘛搞這麼複雜」。
>
> **第一個問題：工具越多，AI 反而越笨。**
>
> 這聽起來很反直覺。工具越多應該越強才對啊？讓我解釋為什麼。

**（投影片顯示 Context Window 示意圖）**

> 你每次跟 AI 對話，它能「記住」的內容是有限的。這個限制叫做 Context Window。Claude 大概有 200K tokens，大約是一本小說的量。聽起來很多對不對？
>
> 但是問題在於——**每一輪對話，AI 都要重新讀取所有工具的定義**。
>
> 我們現在有 55 個工具。每個工具的定義包含名稱、描述、參數格式，大約 300 tokens。
>
> 55 個工具 × 300 tokens = 16,500 tokens
>
> 再加上 CLAUDE.md（AI 的行為指引文件）大約 5,000 tokens。
>
> **也就是說，每一輪對話還沒開始，AI 就已經花掉 21,000 tokens 在讀工具定義。** 這些東西你看不到，但它在背景默默吃掉你的空間。
>
> 隨著工具越來越多，如果到 100 個的話：
>
> 100 個工具 × 300 tokens = 30,000 tokens
> 加上 CLAUDE.md = 35,000 tokens
>
> 200K 的窗口，光工具定義就佔了將近 18%。

> 但真正的問題不只是空間不夠。**更大的問題是：AI 要在 55 個工具裡面選一個來用。**
>
> 你如果是結構工程師，你根本不需要排煙窗檢討工具、不需要帷幕牆面板工具、不需要 MEP 管帽工具。但 AI 每次都要把這些不相關的工具看一遍，然後判斷「這個不是、那個不是」。
>
> 工具越多，AI 選錯的機率越高。有人可能遇過 AI 明明應該用 `query_elements_with_filter` 但它卻用了 `query_elements`，結果查不到想要的東西。這就是工具太多造成的噪音。

> 我問大家一個問題：**你有沒有遇過 AI 選錯工具，或者明明有更好的工具但它沒用？**

（等待回應，收集 1-2 個案例）

> 好，所以第一個問題總結：**工具全量載入 → 佔空間 + 增加噪音 → AI 判斷力下降。**

### 講者備註
- 用「一本小說」比喻 200K tokens，讓非技術人員有感
- Context Window 的概念對初學者很新，可以多花 30 秒解釋
- 互動提問很重要，讓現場不要變成單向講課

## 2-2. 問題二：寫了規範但 AI 不讀（5 分鐘）

### 逐字稿

> **第二個問題：我們辛苦寫好的工作流程，AI 不一定會照做。**
>
> 上次讀書會之後，我們寫了 23 個 Domain 文件。什麼是 Domain 文件？就是把你的專業知識寫成 SOP，告訴 AI「遇到排煙窗檢討應該按什麼步驟來」。
>
> 舉個例子：

**（投影片顯示對比）**

> 假設你跟 AI 說：「幫我檢查 3 樓走廊的排煙窗合不合格」
>
> **情況 A — AI 沒有讀 Domain 文件：**
>
> AI 會自己猜。它可能只是查了窗戶面積然後跟房間面積比一比。但它不知道台灣法規有一條很關鍵的規定：**只有天花板往下 80 公分以內的窗戶面積才算有效**。它也不知道推拉窗只能算一半面積（折減 0.5），固定窗完全不算（折減 0）。
>
> 結果：AI 告訴你「通過」，但其實是不合格的。**這比不檢查還危險，因為你會以為沒問題。**
>
> **情況 B — AI 有讀 Domain 文件：**
>
> Domain 文件裡面寫著：
> - 法規依據：建技規§101①、消防§188③⑦
> - 有效帶：天花板下 80cm
> - 折減係數：推射窗 1.0、推拉窗 0.5、固定窗 0
> - 計算方式：有效窗面積 ÷ 防煙區劃面積 ≥ 2%
> - 呼叫工具：`check_smoke_exhaust_windows`
>
> AI 照著做，結果準確，還能告訴你哪些房間不合格、建議怎麼改。

> 好，問題來了：**為什麼 AI 不讀 Domain 文件？**
>
> 因為 Domain 文件是放在專案的 `domain/` 資料夾裡。AI 不會主動去翻資料夾找檔案來讀。它需要有人告訴它「你現在應該去讀這個檔案」。
>
> 我們之前的做法是在 CLAUDE.md 裡面放一個觸發關鍵字表：

```
| 觸發關鍵字 | 對應 Domain 文件 |
|-----------|----------------|
| 排煙、排煙窗 | domain/smoke-exhaust-review.md |
| 防火、耐燃   | domain/fire-rating-check.md |
| 走廊、逃生   | domain/corridor-analysis-protocol.md |
```

> 但是 CLAUDE.md 本身也在變長。現在已經 300 多行了。AI 對長文件的後半段注意力會衰減——就像你讀一份 30 頁的合約，越後面越不仔細看。
>
> 而且我們有不同的 AI Client — Claude 讀 CLAUDE.md，Gemini 讀 GEMINI.md。每加一個 Domain，兩邊都要更新，管理負擔越來越重。

> 總結第二個問題：**Domain 寫了但沒有可靠的觸發機制 → AI 用自己的想像做事 → 結果不符標準。**

### 講者備註
- 排煙窗的例子很具體，建築師都聽得懂
- 「比不檢查還危險」這句話很有衝擊力，可以稍微停頓讓大家消化
- 可以問「你有沒有遇過 AI 給出看起來合理但其實不對的結果？」

## 2-3. 問題三：多人貢獻時的衝突（5 分鐘）

### 逐字稿

> **第三個問題，是程式碼層面的，但我用非技術的方式來講，每個人都聽得懂。**
>
> 我們這個專案是開源的，歡迎大家一起貢獻。上次讀書會之後有四個人提交了修改，加入了帷幕牆、排煙窗、明細表等功能。
>
> 但是這裡有一個結構性的問題。

**（投影片顯示「共用筆記本」比喻）**

> 想像一下，我們有一本共用筆記本。所有人的工具都寫在這本筆記本的同一頁上。
>
> Alex 要加帷幕牆工具 → 他在第 242 行後面插入他的程式碼
> Alex 也要加排煙窗工具 → 他也在第 242 行後面插入程式碼
> CyberPotato 要加明細表工具 → 他也在第 242 行後面插入程式碼
>
> 三個人都改同一頁的同一個位置。當我要把他們的修改合併在一起的時候——衝突了。Git 不知道誰的應該排在前面，需要我手動一行一行整理。
>
> 而且 Alex 的 PR 包含了 194 個檔案的修改。為什麼？因為他在開發過程中，不小心把編譯產物（bin 資料夾）、暫存腳本、甚至簡報圖片都一起提交了。這不是他的錯——是我們的專案結構沒有幫他把事情隔離開。
>
> **根本原因是：所有功能都擠在兩個檔案裡。**
>
> ```
> MCP/Core/CommandExecutor.cs      ← 3000+ 行，所有 C# 命令都在這裡
> MCP-Server/src/tools/revit-tools.ts ← 900+ 行，所有工具定義都在這裡
> ```
>
> 不管你要加什麼功能，你都得改這兩個檔案。只要有兩個人以上同時在做不同功能，就一定會衝突。

> **Alex 後來自己總結了這個教訓。** 他寫了一份 Git Worktree 開發指南，裡面有一段話我覺得講得很好：
>
> 「錯誤做法：在 main 上直接開發，三個功能混在一起改同一個檔案，改完才想到要拆分成乾淨的 PR，結果花了很多時間用 Worktree 硬拆。」
>
> 他的解法是用 Git Worktree 讓每個功能有自己的資料夾，避免混在一起。這個方向是對的，但只解決了「開發時不會混」的問題，沒有解決「合併時會衝突」的問題——因為最後大家還是改同一個檔案。

> 總結第三個問題：**所有功能擠在同兩個檔案 → 多人同時貢獻必衝突 → 整合成本越來越高。**

# 第三段：Architecture V2 解說（0:25 - 0:50，25 分鐘）

## 段落目的

這是整堂課的核心。要讓參加者理解三層架構的每一層各自做什麼、檔案放在哪裡、Profile 怎麼設定。特別著重在「Skill 和 Domain 的關係」以及「CLAUDE.md 和 GEMINI.md 各自的角色」，因為這是大家最容易搞混的地方。

## 3-1. 三層模組架構總覽（5 分鐘）

### 逐字稿

> 好，接下來是今天最重要的部分。我要介紹我們的新架構，叫做 Architecture V2。
>
> 剛才我們講了三個問題：AI 載入太多工具、Domain 沒被讀、多人改同一個檔案。我們的解法是把系統分成三層，每一層做一件事。
>
> 我先畫一個全貌，然後再一層一層講。

**（投影片顯示三層圖）**

```
使用者: 「幫我檢查 3 樓排煙窗」
         │
         ▼
┌─ 第一層：Skill 觸發 ──────────────────────┐
│                                            │
│  AI 看到 skill.md 的 description           │
│  裡面寫著「排煙」這個關鍵字                   │
│  → 告訴 AI：先去讀 Domain 文件               │
│                                            │
│  成本：~50 tokens（就一行描述文字）            │
└──────────────────┬─────────────────────────┘
                   ▼
┌─ 第二層：Domain SOP ──────────────────────┐
│                                            │
│  完整的法規判定流程                           │
│  step-by-step 告訴 AI 要呼叫哪些工具          │
│  「先呼叫 get_all_levels，再呼叫              │
│   check_smoke_exhaust_windows」             │
│                                            │
│  成本：命中的時候才載入 ~1500 tokens          │
└──────────────────┬─────────────────────────┘
                   ▼
┌─ 第三層：MCP Tools 執行 ──────────────────┐
│                                            │
│  預先開發好的工具                             │
│  AI 只要帶參數呼叫，不用自己寫程式             │
│                                            │
│  成本：只載入該模組的工具 ~2000 tokens         │
└────────────────────────────────────────────┘
```

> 我用一個生活化的比喻來幫大家理解。
>
> 想像你去醫院：
> - **第一層 Skill** = 掛號台。你說「我肚子痛」，掛號台根據關鍵字把你分到「腸胃科」。它不幫你看病，只負責分流。
> - **第二層 Domain** = 醫生的看診 SOP。腸胃科醫生有一套標準流程：先問病史、再做觸診、然後開檢驗單。這就是 Domain 文件——專業知識的標準流程。
> - **第三層 Tools** = 醫療器材。超音波、血液檢查、X 光。醫生按照 SOP 來決定用哪個器材。

> **重點是：沒有 Skill 的觸發，AI 就不會去讀 Domain；沒有 Domain 的引導，AI 就不知道該用哪個工具、用什麼順序。三層缺一不可。**

> 我們來算一下成本差異：
>
> | 方案 | 每輪對話的 Token 成本 |
> |------|---------------------|
> | 舊方案：55 個工具全部載入 + 整份 CLAUDE.md | ~21,000 tokens |
> | 新方案：20 個 Profile 工具 + 命中的 1 個 Domain | ~5,000 tokens |
>
> **省了 4 倍。** 而且 AI 只看到跟自己角色相關的工具，選錯工具的機率大幅下降。

### 講者備註
- 醫院比喻對非技術人員非常有效
- 「三層缺一不可」這句話要強調
- Token 數字不用精確記住，重點是「省 4 倍」這個概念

## 3-2. 第一層細講：Skill 是什麼、放在哪裡（7 分鐘）

### 逐字稿

> 好，接下來我一層一層講。先講第一層：Skill。
>
> **Skill 是什麼？**
>
> Skill 就是一個 Markdown 文字檔，告訴 AI「當使用者提到某些關鍵字的時候，你應該啟動什麼工作流程」。
>
> 它非常輕量——就是一個名字加一段描述。AI 在每次對話開始的時候會掃一遍所有 Skill 的名字和描述（每個大約 50 tokens），如果你的問題命中了某個 Skill 的關鍵字，它就會啟動那個 Skill。

> **Skill 長什麼樣子？**
>
> 我們打開一個實際的 Skill 檔案來看：

**（投影片顯示 fire-safety-check/skill.md 的實際內容）**

```yaml
---
name: fire-safety-check
description: "Execute fire safety compliance checks in Revit:
  fire rating visualization, corridor fire analysis,
  and exterior wall opening distance checks.
  TRIGGER when: user mentions fire rating, 防火, 耐燃,
  corridor, 走廊, Article 45, Article 110.
  Tools: override_element_graphics, check_exterior_wall_openings,
  query_elements_with_filter."
---

# Fire Safety Compliance Check

## Sub-Workflows

### 1. Fire Rating Visualization
Color-code walls by fire rating parameter:
1. get_category_fields for Walls → find fire rating parameter name
2. get_field_values → list all rating values in project
3. query_elements_with_filter → filter walls by rating
4. override_element_graphics → apply colors per rating level

### 2. Corridor Fire Analysis
...

### 3. Exterior Wall Opening Check (Articles 45 & 110)
...
```

> 大家注意幾個重點：
>
> **第一，最上面的 YAML 區塊**（兩條 `---` 之間的部分）。這裡面有兩個欄位：
> - `name`：Skill 的名字
> - `description`：描述文字，**關鍵字就寫在這裡**。AI 會掃這段文字來判斷要不要啟動
>
> 你看它寫了 `TRIGGER when: user mentions fire rating, 防火, 耐燃, corridor, 走廊`。這就是觸發條件——你只要跟 AI 講話時提到「防火」或「走廊」，這個 Skill 就會被啟動。
>
> **第二，YAML 下面的 Markdown 內容。** 這是 Skill 啟動後 AI 會讀到的完整指引。它會告訴 AI：
> - 有哪些子工作流程
> - 每個步驟要呼叫哪個工具
> - 要用什麼順序
>
> **注意：Skill 裡面也可以寫「先去讀某個 Domain 文件」。** 這就是 Skill 和 Domain 的連接方式。

> **Skill 放在哪裡？**

**（投影片顯示資料夾結構）**

```
你的專案根目錄/
├── .claude/
│   └── skills/                    ← 所有 Skill 都放這裡
│       ├── fire-safety-check/
│       │   └── skill.md           ← 一個 Skill = 一個資料夾 + 一個 skill.md
│       ├── building-compliance/
│       │   └── skill.md
│       ├── element-query/
│       │   └── skill.md
│       ├── smoke-exhaust/         ← （規劃中）
│       │   └── skill.md
│       └── ... 共 11 個
```

> **路徑規則：**
> - 固定在專案根目錄的 `.claude/skills/` 下面
> - `.claude` 是一個以點開頭的隱藏資料夾
> - 每個 Skill 有自己的資料夾，裡面放一個 `skill.md`
> - 資料夾名稱就是 Skill 的 ID

> 現在我要講一個很重要的事情。

> **Skill 是 Claude Code 專屬的功能。**
>
> 如果你用的是 Claude Desktop、Claude Code 或 Claude CLI，Skill 會自動生效——AI 啟動時會掃描 `.claude/skills/` 資料夾。
>
> 但如果你用的是 **Gemini CLI**，它不認識 `.claude/skills/`。Gemini 有自己的系統：它讀的是專案根目錄的 **GEMINI.md** 檔案。
>
> 如果你用的是 **VS Code Copilot**，它也不認識 `.claude/skills/`。Copilot 看的是 `.github/copilot-instructions.md` 或者專案根目錄的設定。

> 那怎麼辦？**同一套知識要怎麼讓不同的 AI Client 都能用？**

### 講者備註
- 打開實際的 skill.md 檔案讓大家看到真實內容，比投影片更有說服力
- 「一個 Skill = 一個資料夾 + 一個 skill.md」要反覆強調
- 接下來的 CLAUDE.md vs GEMINI.md 是重點中的重點

## 3-3. CLAUDE.md 和 GEMINI.md 的角色（5 分鐘）

### 逐字稿

> 這個問題我花一點時間講清楚，因為它是大家最容易搞混的地方。
>
> 我們的專案支援多個 AI Client：Claude、Gemini、VS Code Copilot。但每個 Client 讀取「AI 應該怎麼行為」的方式不一樣。

**（投影片顯示對照表）**

```
┌────────────────────────────────────────────────────────┐
│               不同 AI Client 的知識來源                   │
├──────────────┬──────────────┬──────────────────────────┤
│ Claude Code  │ Gemini CLI   │ VS Code Copilot          │
├──────────────┼──────────────┼──────────────────────────┤
│ CLAUDE.md    │ GEMINI.md    │ .github/copilot-         │
│              │              │   instructions.md        │
│ .claude/     │ （沒有對應    │ （沒有對應               │
│  skills/     │  的 skill    │  的 skill                │
│              │  機制）       │  機制）                   │
├──────────────┼──────────────┼──────────────────────────┤
│ domain/      │ domain/      │ domain/                  │
│ （被 Skill   │ （要在        │ （要在 instructions     │
│  觸發後讀取） │  GEMINI.md   │  裡面引導）              │
│              │  裡面引導）   │                          │
├──────────────┼──────────────┼──────────────────────────┤
│ MCP Tools    │ MCP Tools    │ MCP Tools                │
│ （共用）      │ （共用）      │ （共用）                  │
└──────────────┴──────────────┴──────────────────────────┘
```

> 大家看這張表。底層的 **MCP Tools 是共用的**——不管你用 Claude 還是 Gemini，最後都是呼叫同一組工具。**Domain 文件也是共用的**——都是同一份 Markdown。
>
> 差別在最上面的觸發層：
>
> **Claude Code** 有 Skill 機制，它會自動掃描 `.claude/skills/`，根據關鍵字匹配來啟動對應的 Skill，然後 Skill 裡面引導它去讀 Domain。這是最理想的流程。
>
> **Gemini CLI** 沒有 Skill 機制。它的做法是讀 `GEMINI.md` 這個檔案。所以我們需要在 GEMINI.md 裡面寫觸發規則——告訴 Gemini「當使用者提到排煙窗的時候，去讀 domain/smoke-exhaust-review.md」。
>
> 這就像是：Claude 有自動分流的掛號台，而 Gemini 需要你在門口貼一張告示牌「肚子痛請去二樓腸胃科」。

> **實際的做法是這樣的：**
>
> **CLAUDE.md** 的角色：
> - 描述專案架構、Build 指令、部署規則
> - **不需要**寫觸發關鍵字表（因為 Skill 會處理）
> - 保持精簡，專注在「開發者需要知道的事」
>
> **GEMINI.md** 的角色：
> - 和 CLAUDE.md 類似的專案描述
> - **需要額外寫**觸發關鍵字表（因為沒有 Skill 機制）
> - 會比 CLAUDE.md 長一些，因為要兼任觸發層的工作
>
> ```markdown
> # GEMINI.md 裡面需要的觸發表（範例）
>
> ## Domain 觸發規則
> | 使用者提到... | 請先讀取... |
> |-------------|-----------|
> | 排煙、排煙窗、smoke exhaust | domain/smoke-exhaust-review.md |
> | 防火、耐燃、fire rating | domain/fire-rating-check.md |
> | 走廊、逃生、corridor | domain/corridor-analysis-protocol.md |
> | 帷幕牆、curtain wall | domain/curtain-wall-pattern.md |
> ```
>
> 所以如果你是 Gemini 使用者，每次有新的 Domain 文件，你需要同步更新 GEMINI.md 的觸發表。
> 如果你是 Claude 使用者，你只需要建立 `.claude/skills/` 下面的 Skill 檔案。

> **一句話總結：Domain 和 Tools 是共用的，觸發方式因 Client 而異。**

### 講者備註
- 這段是概念密度最高的部分，語速放慢
- 表格一定要投影出來，口頭講容易混亂
- 「掛號台 vs 告示牌」的比喻幫助理解 Skill vs GEMINI.md 的差別
- 如果有人問「那我應該用哪個 Client？」——回答：都可以，Claude Code 的 Skill 體驗最好，但 Gemini CLI 免費且好用

```
domain/
├── smoke-exhaust-review.md        ← 排煙窗法規（建技規§101）
├── fire-rating-check.md           ← 防火時效標準
├── corridor-analysis-protocol.md  ← 走廊寬度檢討
├── daylight-area-check.md         ← 採光檢討（§41）
├── floor-area-review.md           ← 容積率計算
├── parking-clearance-check.md     ← 停車場淨空
├── curtain-wall-pattern.md        ← 帷幕牆排列
├── qa-checklist.md                ← QA 驗證清單
└── ... 共 23 個
```

> Domain 文件的核心結構：
>
> 1. **觸發條件** — 什麼情況下應該讀這份文件
> 2. **法規依據** — 引用的條文和標準
> 3. **檢討步驟** — step by step 告訴 AI 該做什麼
> 4. **呼叫的工具** — 明確列出要用哪些 MCP 工具
>
> **最重要的是第 4 點。** 如果你在 Domain 裡面寫清楚「呼叫 check_smoke_exhaust_windows 這個工具」，AI 就不會自己亂寫程式碼去呼叫 Revit API。它會乖乖使用我們預先開發好的工具。
>
> **這就是三層架構的核心價值：Skill 負責觸發、Domain 負責引導、Tools 負責執行。每一層都不越界。**

| Profile | 載入的工具模組 | 工具數 | 適合誰 |

| `fire-safety` | 基礎 + 房間 + 排煙窗 + 視覺化 | ~25 | 消防審查 |

> **怎麼設定 Profile？**
>
> 在你的 AI Client 設定檔裡面，加一個 `env` 欄位：

```json
{
  "mcpServers": {
    "revit-mcp": {
      "command": "node",
      "args": ["<你的完整路徑>/REVIT_MCP_study/MCP-Server/build/index.js"],
      "env": {
        "MCP_PROFILE": "architect"
      }
    }
  }
}
```

> 就這樣，加一行 `"MCP_PROFILE": "architect"`。AI 就只會看到 35 個跟建築師相關的工具，不會看到 MEP 管帽、排煙窗那些不相關的工具。
>
> **如果你不設定，預設是 `full`，跟以前一樣全部載入。** 所以這是一個純加分的功能，不影響現有使用者。
>
> 這個設定在 Claude Desktop、Gemini CLI、VS Code Copilot 都一樣，因為 `env` 是 MCP 標準格式。

## 4-1. 同步最新版（5 分鐘）

### 逐字稿

> 好，剛才講了很多概念，接下來我們回到實務。你現在手上的版本可能是一月或二月的，我帶大家一起更新到最新版。
>
> 打開你的終端機（PowerShell 或 Git Bash），進到你的專案資料夾：

```bash
cd <你 clone 的位置>\REVIT_MCP_study
```

> **Step 1：確認你有設定 upstream**
>
> upstream 就是我們的「主倉庫」。你 fork 出去的是你的副本，upstream 是大家共用的版本。

```bash
git remote -v
```

> 你應該看到兩個 remote：
> - `origin` → 你自己 fork 的 GitHub 網址
> - `upstream` → `https://github.com/shuotao/REVIT_MCP_study.git`
>
> 如果你沒有看到 `upstream`，跑這個指令加上去：

```bash
git remote add upstream https://github.com/shuotao/REVIT_MCP_study.git
```

> **Step 2：拉取最新版並同步**

```bash
git fetch upstream
git checkout main
git rebase upstream/main
git push origin main --force-with-lease
```

> 我逐行解釋：
> - `git fetch upstream` — 去主倉庫看有沒有新東西（只下載，不合併）
> - `git checkout main` — 切到你的 main 分支
> - `git rebase upstream/main` — 把主倉庫的新 commit 接到你的分支上面
> - `git push origin main --force-with-lease` — 把更新後的 main 推到你的 GitHub fork
>
> **如果 rebase 出現衝突怎麼辦？**
>
> 最常見的原因是你直接在 main 上改過檔案。解法：

```bash
# 先備份你的修改
git stash

# 重新 rebase
git rebase upstream/main

# 把你的修改拿回來
git stash pop
```

> 如果你不確定怎麼處理衝突，最安全的方式是：

```bash
# 放棄 rebase，回到原本的狀態
git rebase --abort

# 改用 reset 直接覆蓋（注意：你在 main 上的修改會消失）
git reset --hard upstream/main
git push origin main --force-with-lease
```

> **Step 3：確認更新成功**

```bash
# 看最新的幾個 commit
git log --oneline -5
```

> 你應該看到類似這樣的內容：
>
> ```
> b35cb89 Merge: Architecture V2 模組化工具系統
> 99620ae Merge: 整合 PR#13 明細表 + MEP 工具
> b0f5ba5 feat: 新增 9 個 domain-based Skills
> ```
>
> 看到 `Architecture V2` 就表示你已經是最新版了。

## 4-3. 常見錯誤排除（7 分鐘）

### 逐字稿

> 接下來是除錯。我整理了大家這幾次最常遇到的問題，一個一個講。

**（投影片顯示錯誤排除表）**

> **問題 1：Revit 載入兩次 MCP 面板**
>
> 症狀：開啟 Revit 後，面板上出現兩個 MCP 按鈕。
>
> 原因：Addins 資料夾裡面有兩個 `.addin` 檔案。一個是正確的 `RevitMCP.addin`，另一個是舊版的 `RevitMCP.2024.addin`。
>
> 解法：

```powershell
# 關掉 Revit

# 看看有幾個 addin 檔案
Get-ChildItem "$env:APPDATA\Autodesk\Revit\Addins\2024\*RevitMCP*"

# 刪掉舊的
Remove-Item "$env:APPDATA\Autodesk\Revit\Addins\2024\RevitMCP.2024.addin"

# 確認只剩一個
Get-ChildItem "$env:APPDATA\Autodesk\Revit\Addins\2024\*RevitMCP*"
```

> 同時也要清理 Git 裡面的舊檔案：

```bash
cd MCP
git rm RevitMCP.2024.csproj 2>$null
git rm RevitMCP.2024.addin 2>$null
```

> **問題 2：dotnet build 報 MSB1011 錯誤**
>
> 症狀：`error MSB1011: Specify which project or solution file to use`
>
> 原因：資料夾裡面有多個 .csproj 檔案，dotnet 不知道要用哪個。
>
> 解法：**指定檔名。**

```powershell
# ❌ 錯誤
dotnet build -c Release.R24

# ✅ 正確
dotnet build -c Release.R24 RevitMCP.csproj
```

> **問題 3：MCP Server 連不上 Revit**
>
> 症狀：AI 說「Connection Timeout」或「Not connected to Revit Plugin」
>
> 排除步驟：
>
> 1. Revit 有開嗎？MCP 面板有按「啟動」嗎？
> 2. Port 一致嗎？Revit 端在 `%AppData%\RevitMCP\config.json` 看 port，MCP Server 預設是 8964
> 3. 如果你改過 port（例如改成 11111），AI Client 設定要加環境變數：

```json
"env": {
  "REVIT_MCP_PORT": "11111"
}
```

> 4. 防火牆有沒有擋？8964 port 要放行 localhost 的連線。

> **問題 4：工具呼叫回傳 NotImplementedException**
>
> 症狀：AI 呼叫 `get_curtain_wall_info` 或 `check_smoke_exhaust_windows` 時出錯。
>
> 原因：這是 PR#11 和 PR#12 的帷幕牆和排煙窗工具。我們已經整合了 TypeScript 端的定義，但 **C# 端的實作還在進行中**（追蹤 Issue #16）。
>
> 目前的狀態：AI 看得到這些工具，但 Revit 端還不能執行。等 Issue #16 完成後就能用了。
>
> **不影響其他工具的使用。** 只有帷幕牆和排煙窗這 13 個新工具暫時不能用，其他 42 個工具都正常。

> **問題 5：npm run build 失敗**
>
> 檢查 Node.js 版本：

```bash
node --version  # 需要 v18 以上
npm --version   # 需要 v8 以上
```

> 如果版本太舊，到 https://nodejs.org 下載 LTS 版本。

- 可以現場問「有沒有人遇過這些問題？」讓遇到的人分享經驗
- NotImplementedException 要坦白講「還沒做完」，不要含糊帶過

## 5-2. Level 1 實作示範：停車場檢討（5 分鐘）

### 逐字稿

> 我用一個完整的例子帶大家走一遍 Level 1 的流程。假設你是建築師，你想讓 AI 能幫你做停車場檢討。
>
> **你需要建立兩個檔案。**

> **第一個：Domain 文件**
>
> 路徑：`domain/parking-clearance-check.md`

```markdown
# 停車場淨空檢討

## 觸發條件
使用者提到：停車場、停車位、車道寬度、parking、淨空

## 法規依據
- 建築技術規則設計施工編 §60、§61、§62
- 標準小型車位：寬 2.5m × 深 5.5m
- 無障礙車位：寬 3.5m × 深 5.5m
- 車道寬度：單車道 ≥ 3.5m、雙車道 ≥ 5.5m

## 檢討步驟
1. 呼叫 get_rooms_by_level 找出停車場空間
2. 呼叫 query_elements_with_filter 篩選停車位族群
3. 讀取每個車位的寬度和深度參數
4. 比對法規尺寸標準
5. 不合規的車位用 override_element_graphics 標示紅色
6. 合規的標示綠色
7. 匯總報告：合規數 / 總數 / 不合規清單

## 注意事項
- 柱子旁的車位可能被柱子佔用淨空，需檢查柱邊距離
- 機械車位的尺寸標準不同，需另行判定
```

> 你看，這就是一份 Markdown 文件。你不需要寫任何程式碼。你只需要把你腦袋裡的知識結構化地寫下來。
>
> **最關鍵的是「檢討步驟」那段**——你要明確寫出要呼叫哪些現有的 MCP 工具。這樣 AI 就不會自己亂來，它會按照你寫的順序呼叫工具。

> **第二個：Skill 觸發定義**
>
> 路徑：`.claude/skills/parking-check/skill.md`

```yaml
---
name: parking-check
description: "停車場淨空檢討：檢查車位尺寸、車道寬度是否符合建技規§60、§61、§62。
  TRIGGER when: 停車場, 停車位, 車道寬度, parking, 淨空, clearance.
  Tools: get_rooms_by_level, query_elements_with_filter,
  override_element_graphics."
---

# 停車場淨空檢討

執行前請先讀取 domain/parking-clearance-check.md 了解法規標準。

## 工作流程
1. 確認目標樓層（通常是 B1F、B2F）
2. 呼叫 get_rooms_by_level 取得停車場空間
3. 呼叫 query_elements_with_filter 篩選停車位
4. 逐一檢查尺寸是否合規
5. 上色標示結果
```

> **就這兩個檔案。** 你寫完之後，提一個 PR 給我們，我們 review 通過就合併。
>
> 合併之後會發生什麼？當任何人跟 AI 說「幫我檢查地下一樓停車場的車位有沒有合規」，AI 就會：
> 1. 看到 Skill 的 description 裡面有「停車場」→ 啟動
> 2. 讀取 domain/parking-clearance-check.md → 知道法規標準和步驟
> 3. 按照步驟呼叫工具 → 得到結果

### 講者備註
- 現場如果可以打開一個文字編輯器即時演示建立這兩個檔案會更有感
- 強調「你不需要寫程式碼」至少兩次
- 停車場是大家都熟悉的題目，很適合做示範
